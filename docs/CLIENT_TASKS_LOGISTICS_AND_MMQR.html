Client checklist: Logistics & MMQR payment

Tasks and decisions you should complete for automatic logistics assignment by delivery area and MMQR payment integration. Open this file in any browser; print or save as PDF if you need a signed copy.

Purpose. Courier rules, pricing, and payment partnerships require your business inputs. This checklist lists only what you need to prepare or decide.

Part A — Logistics (delivery range & auto-assign carrier)

Goal: use customer address (and optionally map coordinates) to determine which logistics company should ship the order, based on rules you define (zones, cities, townships, postal areas, etc.).

A1. Business decisions (you choose)

  • Carriers: List every logistics company you will use (exact legal/trading names). Mark which are active on day one.
  • Assignment strategy: Decide the rule type: e.g. by city/township, by postal code, by distance from warehouse, or a priority list (fallback if the first carrier does not serve that area).
  • Who pays shipping: Customer only / seller absorbs / mixed (e.g. free above a cart total). Any cap or flat rates per zone.
  • Fallback: What should happen if no carrier matches the address (block checkout vs. “manual quote” vs. default carrier).

A2. Data & documents you must deliver

  • Zone matrix: Spreadsheet (CSV/Excel), e.g. Region → City/Township → Assigned carrier code → Notes. Include all exceptions (remote areas, islands, etc.).
  • Address model: Confirm which fields customers must fill (Myanmar: region/state, city, township, address line, phone). If you use a standard list (e.g. fixed township dropdowns), provide the master list or approve using a public reference dataset.
  • Carrier APIs or process: For each carrier, either official API documentation (sandbox + production), partner portal login instructions, or a clear statement that assignment is rule-only (no live API) and staff will book shipments outside the app.
  • Credentials / test access: Sandbox keys, merchant IDs, webhook URLs (if carriers push status updates). Share through a secure channel (avoid plain email when possible).
  • Labels & compliance: Any export, restricted goods, or COD rules that change carrier choice.

A3. Approvals

  • Sign off on a single source of truth for the zone table (who may edit it later: admin UI vs. only your team).
  • Approve test scenarios (sample addresses that must map to carrier A vs. B) before go-live.

Optional — extra detail (if you have it)

  • Sample 5–10 real-looking addresses (anonymized) and the carrier you expect for each.
  • Whether you want map / geocoding (e.g. Google Maps, Mapbox) — note any budget for map API usage.

Part B — MMQR payment integration

“MMQR” usually refers to Myanmar QR payment rails used by banks and mobile wallets. The exact integration depends on which institution or aggregator you contract with. The product must match their specification (QR payload, merchant ID, callback, settlement).

B1. Business decisions (you choose)

  • Provider: Name the bank, wallet, or payment aggregator (PSP) that will issue your MMQR and settle funds (e.g. specific Myanmar bank PSP, or a licensed aggregator — use the exact entity name).
  • Channels: Confirm whether customers pay only via QR scan in-app / at checkout, or also in-store / manual upload of slip.
  • Currency & settlement: MMK only? Settlement cycle (T+1, etc.) and which business bank account receives payouts.
  • Refunds & disputes: Who approves refunds, and whether the PSP supports API refunds or manual process.

B2. What you must obtain from the PSP / bank

  • Written integration pack: API spec, QR generation rules (EMV / MMQR field layout), sample payloads, error codes.
  • Sandbox / UAT environment with test merchant credentials and test customer wallets if applicable.
  • Production onboarding checklist from them: KYC, merchant ID, terminal ID (if any), IP allowlisting, webhook/callback URLs you must register.
  • Contact: Technical support contact for integration issues (name, email, phone, hours).

B3. Your deliverables (documents & access)

  • Decision record: Chosen PSP name + product name (e.g. specific MMQR product) + link to official docs if public.
  • Credentials: Sandbox keys first; production keys later via a secure handoff (avoid sharing secrets in open chat).
  • Website domains: Confirm base domain(s) for production and staging (e.g. https://yourdomain.com) for registering callbacks with the PSP.
  • Payment rules you want: When an order counts as “paid” (e.g. instant confirmation vs. manual check). What happens if the customer abandons the QR flow.
  • Legal: Confirmation that your business is authorized to accept payments through that PSP under your license / registration.
